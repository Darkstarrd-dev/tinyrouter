## 二、总体结论与发布就绪度

### 2.1 问题计数（去重后）

| 严重度 | 数量 | 说明 |
|---|---|---|
| **Critical** | 1 | `os.Exit(0)` 在 `Shutdown` 末尾绕过 defer/return |
| **High** | 21 | 19 个活跃问题 + 2 个覆盖率高危（复审后 H-5/webview/OpenBrowser 三项降级 Medium） |
| **Medium** | ~53 | 含正确性边界、覆盖缺口、文档脱节 |
| **Low** | ~30 | 风格/加固/文档同步 |

### 2.2 发布就绪度评估

**结论：当前状态不建议直接打 v1.8.0 tag。** 应先修复下方"发布前必修清单"中的 Critical + 高优先级 High（约 8-10 项），其余 High 可在 v1.8.0 patch 或 v1.8.1 跟进。

主要风险集中在四类：
1. **配置热重载失效**（`rt.cfg` stale 指针，影响下载目录、monitor 白名单、CSP 端口）。
2. **HTTP 契约/会话安全**（非流式 200/502 状态码错乱、cookie 缺 `Secure`、CORS 反射任意 Origin）。
3. **前端资源生命周期**（多处 blob URL 与 SSE/WebSocket/listener 不释放，长会话内存累积）。
4. **gallery（v1.8.0 核心新功能）**：TIFF 解码 DoS、session ID 失败回退常数、imageProxy SSRF。

亮点：AES-GCM 密码学实现严谨、密钥不落 state.yaml、登录限流 + 常量时间比较、注入面整体隔离良好（全部 `exec.CommandContext` 参数数组、无 shell 拼接）、转义纪律与 DOMPurify 净化到位。

## 三、Critical 级发现（必修）

### C-1. `Shutdown` 末尾 `os.Exit(0)` 绕过 defer/return，破坏优雅关闭语义
- **位置**：`internal/app/app.go:276`（`os.Exit(0)`）、`app.go:244-246`（`Run` 中 `ctx, cancel := WithTimeout(..., 10s); defer cancel(); return a.Shutdown(ctx)`）、`app.go:278`（`return nil` 为死代码）
- **描述**：`Shutdown` 在 `sm.Shutdown(ctx)` / `stateManager.FlushSync()` / `apiRouter.Cleanup()` / `lockFile` 清理后调用 `os.Exit(0)`。该调用：
  - 跳过 `Run` 中所有 `defer`（虽 `defer cancel()` 无害，但破坏 Go 正常返回/defer 纪律）；
  - 使 `return nil`（行 278）成为死代码；
  - 对 webview/tray 变体是"防僵尸进程"的有意设计，但对 default/console 变体属不必要硬退出；正在进行的 SSE/下载流在 10s drain 窗口外被硬切断；
  - `Shutdown` 的 error 返回值（`forced shutdown: %w`）在 os.Exit 路径下永远不可达 `main.go` 用于日志。
- **影响**：发布质量层面的架构性缺陷；webview 变体下偶发 COM/WebView2 runtime 状态未释放。已抽查核验：cleanup 确实在 os.Exit 之前执行（非"完全不清理"），但绕过 defer/return 的反模式对维护与回归保护不利。
- **建议**：仅在 webview/tray 变体检测到消息循环未退出时才 `os.Exit`；default/console 直接 `return`。可用 build-tag 层 `forceExitIfNeeded()`（默认空实现，tray/webview windows 填充）。补 `Shutdown` 可达性单测。

## 四、High 级发现（强烈建议发布前修）

按主题分组。标注"⭐"的为发布前最高优先级。

### 4.1 安全 / 鉴权 / 网络边界

#### H-1 ⭐. `rt.cfg` 指针 reload 后永不刷新，handler 长期读取 stale 配置（含 monitor 白名单）
- **位置**：`internal/api/router.go:77-97`（`New` 注入 `deps.cfg`）、`registry/registry.go:73-78`（`Reload` 仅替换内部指针）、`internal/api/download.go:42,134`（读 `rt.cfg.Download.DefaultDir`）、`internal/api/monitor.go:37`（读 `rt.cfg.Monitor.AllowedCommands`，**安全相关**）、`internal/api/router.go:175`（`securityHeaders(rt.cfg.Port)` 闭包捕获端口）
- **描述**：`Router.deps.cfg` 是 `*config.Config`，仅在 `New()` 注入一次。所有 reload 路径（`updateSettings`/`reload`/各 CRUD）只调 `reg.Reload(cfg)` 替换 registry 内部指针，**不回写 `rt.cfg`**。而 `download.go`/`monitor.go` 直接读 `rt.cfg` 而非 `rt.reg.Config()` 快照。
- **影响**：用户经 `/api/settings` 或 `/api/reload` 改配置后：下载默认目录仍用旧值；monitor 白名单不生效（收紧后仍按旧白名单放行——配置型安全漏洞）；CSP `connect-src ws://127.0.0.1:<Port>` 用旧端口（端口变更后 WebSocket 被 CSP 拦截）。直到进程重启才恢复。
- **建议**：移除 `deps.cfg` 字段，统一改走 `rt.reg.Config()` 快照；或 `reg.Reload` 后由 app 层同步回写 `rt.cfg`（需加锁）。最小改动：将 `download.go:42/134`、`monitor.go:37`、`router.go:175` 改为读 `rt.reg.Config()`。

#### H-2 ⭐. Session cookie 未置 `Secure` 属性（三处子评审交叉确认）
- **位置**：`internal/api/auth.go:163-171`（`setSessionCookie`）、`auth.go:151-158`（登出 cookie）
- **描述**：cookie 设了 `HttpOnly` 与 `SameSite=Strict`，但**无 `Secure`**。同时 `MaxAge=86400*30`（30 天）与服务端 `sessionMaxAge=24h` 不一致。
- **影响**：若经反代/HTTPS 暴露，会话 token 可在 HTTP 明文链路被嗅探；30 天客户端存活窗口在被窃取场景下延长暴露面。纯 localhost 单机下实际风险低，但违反"登录态安全"最佳实践。
- **建议**：设 `Secure: true`（现代浏览器对 localhost 允许，不破坏本地使用）；将 `MaxAge` 与 `sessionMaxAge`（24h）对齐或在文档明确语义。

#### H-3 ⭐. `/v1/*` CORS 预检反射任意 Origin
- **位置**：`internal/api/router.go:183-193`
- **描述**：`r.Options("/v1/*", ...)` 在请求带 `Origin` 时直接 `Set("Access-Control-Allow-Origin", origin)` + `Vary: Origin`，未校验 Origin 是否为 localhost/受信来源。
- **影响**：任意外部网页可在用户浏览器内对 `http://127.0.0.1:<port>/v1/*` 发起跨域请求（虽未设 `Allow-Credentials`，无法窃 cookie，但可探测/滥用本地代理、消耗上游配额）。
- **建议**：仅允许 `http://127.0.0.1:<port>` / `http://localhost:<port>` 本地 Origin；或干脆不对 `/v1/*` 开放跨域（前端同源调用本就不需要 CORS）。

#### H-4 ⭐. imageProxy / saveImage 服务端请求伪造（SSRF，无私网地址拦截）
- **位置**：`internal/api/image.go:122-150`（`imageProxy`）、`image.go:20-117`（`saveImage`）、`router.go:91`（`testClient` 无 `CheckRedirect` 限制）、`url_validation.go:9-27`（注释明确不拦截私网/回环）
- **描述**：`imageProxy` 仅校验 `http://`/`https://` 前缀即由服务端 `rt.testClient.Get(u)` 抓取任意 URL 并流式回传浏览器。无内网/回环/云元数据（`169.254.169.254`）地址过滤。三处子评审交叉确认。
- **影响**：已通过 admin 鉴权的本地页面/恶意网页可借 TinyRouter 进程身份探测内网、读取云元数据（返回内容显示在管理 UI）。
- **建议**：对目标 host 做 IP 解析后拒绝 `127/8`、`169.254/16`、`::1`、链路本地、RFC1918 私网；设置 `testClient.CheckRedirect` 限制跳转；至少在文档/发布说明中明确"纯本地工具不具备内网隔离"。

> 注：原 H-5（saveImage 扩展名）经复审降级为 Medium——`imgs/` 目录无 HTTP 路由回读（全仓 grep 确认 FileServer 仅服务 embed.FS），存储型 XSS 当前不可达。详见 §五 Medium 表"api (image)"行。编号保留不再复用，故本节 H-4 之后为 H-6。

#### H-6. download 客户端可任意指定 `downloadDir`，无路径白名单/越界校验
- **位置**：`internal/api/download.go:41-42,133-134`、`internal/download/manager.go:272-273`、`internal/download/args.go:84-89`
- **描述**：`createDownload`/`createPlaylistDownload` 直接信任请求体 `input.DownloadDir`，未做校验；`BuildDownloadArgs` 经 `filepath.Join(dir, template)` 拼入 `-o` 传给 yt-dlp。攻击者可提交 `downloadDir: "/"` 或 `"../../../../etc"` 写文件到任意目录。
- **影响**：任意目录写文件（覆盖系统文件、写入启动项）。纯本地但同机恶意网页/多用户可达。
- **建议**：API 层校验 `DownloadDir` 必须非空且位于允许根目录（如 `DefaultDir` 子树）内，`filepath.Clean` + 前缀校验，拒绝 `..` 与绝对路径越界。

#### H-7. download URL 完全透传，无协议/主机白名单（SSRF / file:// 跳板）
- **位置**：`internal/api/download.go:28-31,64-67,89-92,120-123`、`internal/download/executor.go:52`
- **描述**：请求体 `url` 仅校验非空，`Execute` 直接把 URL 透传给 yt-dlp。yt-dlp 支持 `http`/`https`/`rtsp`/`file://` 等。客户端可传 `file:///etc/passwd` 或内网地址，借本地进程发起请求/读取。元数据探测接口 `getVideoInfo`/`getPlaylistInfo` 同样透传并回显。
- **影响**：SSRF / 本地文件读取跳板。
- **建议**：限制 scheme 为 `http`/`https`（必要时 `magnet`/`rtmp`），拒绝 `file://` 与环回/链路本地/私网地址；元数据接口同样收紧。

### 4.2 正确性 / 契约

#### H-8 ⭐. 非流式 keep-alive 提前 flush 导致"全 key 耗尽"返回 HTTP 200 而非 502
- **位置**：`internal/proxy/forward.go:226-237`（提前 `w.Write("\n")`+`Flush` 触发隐式 `WriteHeader(200)`）、`forward.go:97-99`（`writeError(w, 502, "all keys exhausted")` 在 headers 已发送后失效）、`docs/proxy-architecture.md:514`（契约写明返回 502）
- **描述**：对任意非流式请求，`forwardWithRetry` 在**第一次迭代、尚未知道上游是否会成功**时就执行 `w.Write([]byte("\n"))` + `flusher.Flush()`。Go 的 `http.ResponseWriter` 在首次 `Write` 时隐式提交 200。若所有 key 最终失败，`writeError(w, 502, ...)` 的 `WriteHeader(502)` 因 header 已提交而被忽略（"superfluous WriteHeader call"），客户端实际收到 `HTTP 200` + `{"error":{"message":"all keys exhausted"}}`。
- **影响**：非流式请求在全部 key 耗尽或最终上游错误时，对外返回错误语义的 200 而非 502/上游状态码。任何按 HTTP 状态判断的下游客户端（SDK、网关、监控）会误判成功。已抽查核验。该 bug 由 keep-alive 特性（§8.7）引入，与文档 §13.1 约定的 502 契约冲突。`handler_test.go:533-535` 把此非预期行为固化为例，掩盖 bug。
- **建议**：keep-alive 字节仅在已确定成功后才开始写入；或非流式路径用缓冲 `ResponseWriter`，仅在上游 2xx 时整体 flush。修复 `TestAfterMaxRetries_WithMock` 断言为真实状态码。

#### H-9. `updateKey` 不区分"未传字段"导致明文 Key 被误清空/覆盖
- **位置**：`internal/api/keys.go:59-78`、`internal/registry/keys.go:73-92`
- **描述**：`updateKey` 将整个 `config.Key` 反序列化后直接 `UpdateKey`，后者无条件覆盖 `k.Key = updates.Key`。前端若只更新 `isActive`/`name`/`priority` 而不传 `key`，`updates.Key` 为空字符串，会**把现有明文密钥覆盖成空**，导致该 key 失效且密钥丢失。
- **影响**：UI 切换 key 启用状态即可能无声清空密钥；数据损坏且无法恢复。
- **建议**：`UpdateKey` 采用 partial update 语义（仅当 `updates.Key != ""` 时覆盖）；或 handler 层做字段级 merge。

#### H-10. gallery session ID 随机数失败回退为固定常数（全会话碰撞）
- **位置**：`internal/api/gallery.go:20-26`
- **描述**：`newGallerySessionID` 在 `rand.Read` 失败时返回 `hex.EncodeToString([]byte("gallery"))`，即固定串 `"67616c6c657279"`。一旦 `crypto/rand` 失败（极少见但属错误路径），**所有**上传的 zip 都会落到同一 sessionId；`put` 不检测碰撞，后一次覆盖前一次，两个并发用户互相串台/读到错误图片。
- **影响**：错误路径下数据损坏与跨用户内容混淆。
- **建议**：失败回退应返回错误让 handler 返回 500，而非用常数；或在 sessionId 中加入时间戳/自增计数；`put` 对相同 id 做碰撞处理。

#### H-22. `version.go` 与发布版本号脱节（1.7.0 vs v1.8.0）
- **位置**：`internal/app/version.go:4`（`const Version = "1.7.0"`）
- **描述**：本次为 v1.8.0 发布评审，但 `Version` 常量仍为 `1.7.0`；最新 git tag 为 `v1.7.8`。启动日志与 About 页会显示错误版本号。
- **影响**：发布核验与用户报障受影响。
- **建议**：发布前将 `Version` 更新为 `1.8.0`（或接入 ldflag `-X` 注入）。

### 4.3 资源泄漏 / DoS

#### H-11 ⭐. 后端 TIFF 解码无像素尺寸上限（内存 DoS）
- **位置**：`internal/gallery/tiff.go:15-19`、`internal/api/gallery.go:104-124`
- **描述**：`galleryConvertTiff` 用 `MaxBytesReader` 限制入参 50MB，但 `tiff.Decode` 将 TIFF 解码为完整 `image.Image`，其内存 = width×height×4 字节，不受 50MB 入参约束。一个宽高极大但压缩率高的 TIFF（如 50000×50000）解码后约 10GB；`jpeg.Encode` 还会再分配缓冲。
- **影响**：单个体积不大的恶意/异常 TIFF 即可造成服务端 OOM。v1.8.0 核心功能建议从严。
- **建议**：解码后校验 `b.Bounds()` 尺寸（如限制 ≤ 16384×16384 或面积阈值），超限返回 400。

#### H-12 ⭐. 前端 object URL 永不 revoke（gallery）
- **位置**：`web/playground/static-pg/gallery.js:185-192`（`trackURL`/`clearObjectURLs`）、`gallery.js:239,262,264`
- **描述**：`trackURL(url)` 把每个 `URL.createObjectURL` 结果压入 `state.objectURLs`，但全程没有 `URL.revokeObjectURL` 调用。`clearObjectURLs()` 只做 `state.mainURL = null`，从不 revoke。切换图片/zip/视频、自动播放切换时，旧 blob URL 持续累积且无法回收。
- **影响**：长时间浏览大图/多 zip 后内存与浏览器 blob 句柄持续增长，可能触发 `OutOfMemoryError`/渲染进程崩溃。
- **建议**：替换 `item.mainURL`/`item.thumbURL` 前 revoke 旧值；`clearObjectURLs` 遍历 `state.objectURLs` 全部 revoke；封装 `setMainURL(item,url)` 统一管生命周期。

#### H-13. Admin UI：Console/Monitor/Terminal 三路流在整页导航离开时未关闭
- **位置**：`web/static/app.js:23-41`（`navigateTo` 仅关 `downloadEventSource` 与 usage 流）、`console.js:163-185`、`monitor.js:63-92`、`terminal.js:227-238`
- **描述**：`navigateTo` 切换到非 Console 页时，未关闭 `consoleEventSource`/`monitorEventSource`/`terminalWebSocket`，也无 `beforeunload`/`pagehide` 兜底。注：Console **子标签内切换**（`switchConsoleTab`）有对应清理钩子，泄漏仅发生在**整页离开**（navigateTo 跳走）路径。
- **影响**：每次离开 Console 页都泄漏 1+ 常驻 SSE + 1 个 WebSocket（直到 tab 关闭），累积服务端 goroutine/连接与浏览器连接，日志流持续推送到已脱离 DOM 的容器。
- **建议**：在 `console.js`/`monitor.js`/`terminal.js` 暴露清理函数，于 `navigateTo` 离开 console 页时统一调用；或 `beforeunload` 兜底。

#### H-14. Admin UI：终端子视图切换不关闭 WebSocket 与 resize 监听器
- **位置**：`web/static/terminal.js:248`（`cleanupTerminal()` 定义，仅 `detachTerminalView()`）、`console.js:99`（`switchConsoleTab` 中的调用点）、`terminal.js:99`（`window.addEventListener('resize', …)`）、`terminal.js:227-238`（`closeTerminalSession` 才 `removeEventListener`+`close`）
- **描述**：`switchConsoleTab` 从 terminal 切到 monitor/logs 时 `cleanupTerminal()` 只摘下 xterm DOM 节点，**未**调 `closeTerminalSession()`。`terminalWebSocket` 保持 OPEN，`window resize` 监听器持续注册。再次进入 Terminal 各 `new WebSocket` + `addEventListener` 一次。
- **影响**：Console 子视图间反复切换累积多个 WebSocket 与 resize 监听器，内存与连接泄漏。
- **建议**：`cleanupTerminal()` 改为调 `closeTerminalSession()`；或 detach 时 `removeEventListener` + `terminalWebSocket.close()`，并在 `initTerminal` 前先 `closeTerminalSession()` 防重。

#### H-15. Playground：HTML 预览 iframe `sandbox="allow-same-origin"` 暴露同源 DOM/凭据可达性
- **位置**：`web/playground/static-pg/pg-render.js:163-187`（`pgRenderHtmlPreview`），尤其 `pg-render.js:172`
- **描述**：模型输出含 ```` ```html ```` / `<!DOCTYPE` / `<svg` / `<?xml` 代码块时，原样经 `iframe.srcdoc` 渲染，sandbox **仅** `allow-same-origin`、无 `allow-scripts`。脚本不会执行（符合"不允许脚本执行"），但 `allow-same-origin` 使 iframe 与管理 UI **同源**，model HTML 可执行同源 `fetch('/api/...')`、读 `document.cookie`、读取 DOM 内明文 API Key 等。配合 `connect-src 'self'` CSP 该请求被放行。
- **影响**：恶意/被投毒模型输出可静默读取同 origin 的管理态信息（Key、Usage、配置）。
- **建议**：HTML 预览 iframe 应使用 `sandbox=""`（完全隔离），或仅在 `load` 后读 `scrollHeight` 即可，无需持续同源。最低限度去掉 `allow-same-origin`。

#### H-16. Playground：`pgInitImageZoom` 每次打开图片泄漏 3 个 window 级 listener
- **位置**：`web/playground/static-pg/pg-modal.js:190,223,230`（`window.addEventListener` resize/mousemove/mouseup）
- **描述**：每次打开图片预览都向 `window` 注册 `resize`/`mousemove`/`mouseup` 监听，且**从不移除**。`body` 上的 `wheel`/`mousedown` 随 modal `innerHTML` 重建回收，但 `window` 级监听永久挂留。
- **影响**：反复打开多张图片后，mousemove/resize 回调持续累积，内存泄漏与无谓的每鼠标移动执行开销。
- **建议**：用命名函数引用在 `pgCloseModal` 时 `removeEventListener`；或 scoped 到 modal body 而非 window。

#### H-17. Playground：`pgOpenMermaidSvg` 生成的 blob URL 永不 revoke
- **位置**：`web/playground/static-pg/pg-render.js:154-161`
- **描述**：点击 Mermaid 占位符时 `URL.createObjectURL(blob)` + `window.open(url, '_blank')`，从不 `revokeObjectURL`，每次都新建 blob 与新标签。
- **影响**：每点击一次 Mermaid 图泄漏一个 blob: 内存对象。
- **建议**：`window.open` 后延迟 `revokeObjectURL`（如 `setTimeout(…, 1000)`）；或改在当前页 modal 内展示 SVG。

#### H-18. Terminal：会话 cleanup 不等待 `pty.Close()`，快速重连可能堆积未释放 PTY
- **位置**：`internal/terminal/session.go:187-193`（`go func(p pty.Pty){ _ = p.Close() }(s.pty)`）
- **描述**：为避免 Windows ConPTY `ClosePseudoConsole` 死锁，`pty.Close()` 放进后台 goroutine 且**不等待**。会话 cleanup 立即返回，`onClose` 把 `activeTerm` 置 nil，第二个会话可立即建立，前一个 PTY 的 Close 仍在后台。
- **影响**：极端快速"断开→重连"场景下，旧 PTY 句柄/ConPTY 资源延迟释放，存在句柄泄漏窗口；若 `p.Close()` 异步 panic（未 recover），goroutine 崩溃。
- **建议**：goroutine 内 `defer recover()` 包裹 `p.Close()`；或带超时的同步 Close。

#### H-19. Unix 进程树杀仅 SIGTERM，无 SIGKILL 兜底（download + monitor 共性）
- **位置**：`internal/download/kill_unix.go:12-18`（仅 SIGTERM）、`internal/monitor/manager_unix.go:15`（仅 SIGTERM）
- **描述**：download 与 monitor 的 Unix 进程清理都只发 SIGTERM，无后续 SIGKILL 兜底。yt-dlp/ffmpeg 在处理大文件或忽略 SIGTERM 时不会立即退出，导致 `cmd.Wait()` 阻塞超时、或 worker 已返回但进程仍在写 `.part` 文件。Monitor 的进程回收由 `Start` 内的 wait goroutine 承担（`manager.go:121`），`Stop` 自身仅 `cancel()` + `killProcessGroup` 不等待——若 SIGTERM 被忽略且 ctx 取消未生效，则 wait goroutine 永不返回、`running` 永为 true，Manager 卡死（再 `Start` 报 already running）。
- **影响**：取消/停机后子进程残留，占用 CPU/带宽、写残留 `.part`；Monitor SIGTERM 被忽略即卡死。
- **建议**：先 SIGTERM，启动 goroutine 等待 grace period（如 2s），超时后 SIGKILL（Unix）/`taskkill /F`（Windows 已是 /F）。Monitor 的 `Stop` 应保证进程回收。

### 4.4 覆盖率高危

#### H-20. terminal 包 20.4% 覆盖率，双向泵/cleanup/单会话守卫零单测
- **位置**：`internal/terminal/session.go`（`NewSession`/`readFromPTY`/`readFromWebSocket`/`waitForProcess`/`cleanup`）、`session_test.go` 全篇
- **描述**：5 个测试仅验证 `defaultShell` 非空与 `killProcessGroup` 不 panic。PTY/WS 双向数据流、resize 协议（5 字节 `0x01`）解析、cleanup 进程树杀、`Close` 幂等触发链完全无自动化验证。
- **影响**：交互式 shell 是 v1.8.0 的 debug 特性，核心双向流无回归保护，发布风险高。
- **建议**：补真实 PTY 启动 `echo` 类命令的端到端测试、resize 消息喂入测试、`NewSession` 失败路径清理断言、`cleanup` 幂等与 `onClose` 调用。

#### H-21. app 包核心路径（New/Run/Shutdown/lock/state 恢复/version）零单测
- **位置**：`internal/app/app.go`、`host.go`、`lock_*.go`、`version.go`、`browser*.go`
- **描述**：仅 `server_manager_test.go` 覆盖 16.5%。`New`（单实例锁）、`Run`（state 恢复、server 启动、host loop）、`Shutdown`（含 C-1 的 `os.Exit`）、`tryLockFile`、`OpenBrowser`、版本常量全部无测试。
- **影响**：发布前最关键的启动/关闭/单实例逻辑缺乏回归保护。
- **建议**：补 `New` 双启动失败、`Shutdown` 可达性（mock SM 验证）、`tryLockFile` 双进程行为。

## 六、跨包 / 横向主题

### 6.1 配置热重载一致性（根因：`rt.cfg` stale 指针）
H-1 的根因是同时存在两套配置访问路径：`rt.reg.Config()`（快照，正确）和 `rt.cfg`（指针，stale）。当前已有 `download.go`/`monitor.go`/`router.go` 三处误用，且新增 handler 极易再踩坑。**这是 v1.8.0 最值得一次性彻底修的架构性问题**——移除 `deps.cfg`，所有读取统一走 `rt.reg.Config()`，可同时消除 H-1 及其衍生 Medium。

### 6.2 前端资源生命周期（共性）
H-12/H-13/H-14/H-16/H-17 共同指向"前端对 blob URL / SSE / WebSocket / window listener 的释放纪律缺失"。建议作为 v1.8.0 前端打磨的一个统一专项：建立"打开即注册、关闭即释放"的封装（如 `setMainURL`/`pgScopedListener`），并在 `navigateTo`/`cleanupPlayground`/modal close 三个边界统一清理。

### 6.3 进程清理一致性（download + monitor + terminal）
H-19（download/monitor 仅 SIGTERM）与 terminal 的 SIGKILL 策略不对称。建议统一为"SIGTERM → grace period → SIGKILL"的跨平台抽象，并保证 `Stop`/`cleanup` 后进程确定性回收。

### 6.4 SSRF 面收敛（跨包）
H-4（imageProxy/saveImage）、H-7（download URL）、`url_validation.go` 显式不拦截私网——三处共同构成"纯本地工具"的 SSRF 面。当前安全边界完全依赖 localhost 绑定。建议至少在发布说明中显式告知用户"不要将本服务暴露于非 localhost"，并对 imageProxy/download 元数据接口补私网地址拒绝名单。

### 6.5 文档同步缺口（AGENTS.md 强制要求）
多处文档与代码脱节（已分别在 Medium 中列出），主要集中在：
- `docs/combo-architecture.md` 行号锚点全面漂移、ID/Name 混用；
- `docs/rotation-architecture.md` §5/§6.2 NIM 过滤口径与 failover tie-break 描述过时；
- `docs/playground-architecture.md §16` LRU 容量/缩略图参数与代码不符；
- `docs/terminal-monitor-architecture.md §4.2.5` 环境变量块描述与 `os.Environ()` 不符；
- `docs/build-variants.md` "最后核对 2026-07-14" 未反映本次变更；
- `PROJECT_MAP.md` 若列 `selector_hot_reload.go` 则需改为 `settings.go`。

按 AGENTS.md「文档同步指令（强制）」，发布前应同步修订。

## 十一、亮点（安全与工程得当之处）

1. **密码学严谨**：AES-256-GCM、nonce 每次 `crypto/rand` 随机、无 `math/rand`、无硬编码密钥、`TestEncrypt_DifferentNonces` 验证随机性。
2. **密钥不落 state.yaml**：`KeySnapshot` 仅持久化冷却/轮转运行时状态，不含 Key 字段。
3. **登录链路扎实**：session token 用 `crypto/rand` 256-bit、`subtle.ConstantTimeCompare` 防时序、登录限流 5 次/分钟 + 封禁、`SameSite=Strict`、过期会话惰性清理。
4. **注入面整体隔离**：yt-dlp/ffmpeg/monitor/terminal 全部 `exec.CommandContext(args...)` 参数数组，无 shell 拼接；monitor 白名单不含任何 shell 二进制。
5. **CORS 严格分区**：`/api/*` 管理路由无 CORS，阻断外部页跨源读取配置/密钥。
6. **转义纪律与 DOMPurify 净化**：admin UI 与 playground 对模型/上游输出统一 `escapeHtml`/`escapeAttr`/`escapeForJsString`，Markdown 管线经 `marked`+`DOMPurify`，Sources 链接经 `pgSafeHref` 协议过滤。
7. **SSE/并发模型规范**：fan-out 广播 + 慢订阅者非阻塞丢弃 + `defer unsub` 成对 + `navGen`/`session++` 离页守卫，避免乱序回调。
8. **原子写契约一致**：config 与 state 共用 `.tmp + rename + 直写回退`，Windows 锁文件场景保留 .tmp 供下次重试。
9. **gallery 不写盘**：zip 字节常驻内存按索引/清洗名匹配，当前无经典 zip-slip 任意文件写出（但 `cleanZipPath` 仍建议规范化 `../` 防未来演进踩坑）。
10. **测试质量高的包**：console（97.9%）、monitor（88.6%）、usage（84.8%）、proxy（70.5%）、rotation（77.7%）的单测设计扎实，含并发、边界、跨日 CST、NIM 阶梯等针对性用例。
