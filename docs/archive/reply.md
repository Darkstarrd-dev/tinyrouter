# TinyRouter 自动对话（群聊）功能：架构审查与指导

先说总体评价：独立迭代模型的方向是对的，abortFlag 抑制 hook、inbox 合并、延迟期间继续累积消息这些细节都处理得很成熟。下面分三部分：**一个核心架构建议**、**对 6 个问题的逐条回应**、**一些能让功能"活起来"的交互灵感**。

---

## 一、核心架构建议：从"广播 + 每窗口 inbox"转向"共享时间线（Shared Transcript）"

这是最重要的一条，它能同时解决你列出的问题 1、2、3 的一半和问题 6。

### 现状的根本问题

当前每个窗口维护自己的 `messages` 数组，消息通过广播复制到各窗口。这意味着：

- **同一条消息在系统中存在 N 份拷贝**，去重靠 `sender + content + timestamp` 三元组（脆弱）
- **各窗口的历史会逐渐分叉**：合并时机不同，导致 A 窗口看到的对话顺序可能与 B 窗口不同
- **群聊 Modal 是"重建"出来的视图**，而不是天然存在的事实

### 建议的模型

把架构反过来：**群聊时间线是唯一事实来源（single source of truth），各窗口的 `messages` 是从时间线派生出的视图**。

```
pgState.autoChat.timeline = [
  { id, sender, senderType: 'user'|'agent', winIdx, content, ts, status }
]

每窗口只需记录: lastReadTimelineId  （读到哪了）
```

窗口要发送请求时，做一次"**视角渲染**"（perspective rendering）：

- 遍历 timeline，**自己发的消息 → `role: assistant`（不带前缀）**
- 其他人发的消息 → 合并为 `role: user`，带 `[sender]:` 前缀
- 从 `lastReadTimelineId` 之后的消息就是"新消息"，替代 inbox 的概念

好处立竿见影：

1. **去重问题消失**——消息只有一份，Modal 直接渲染 timeline，不需要三元组去重
2. **inbox 数据结构消失**——"有没有新消息"变成 `lastReadTimelineId < timeline 最新 id` 的比较，状态更少、竞态更少
3. **上下文管理有了统一抓手**——截断/摘要只需对 timeline 做一次，所有窗口受益（见下）
4. **顺序一致性天然保证**——所有窗口看到同一个时间线，只是视角（role 映射）不同
5. **为后续功能铺路**——导出剧本、回放、导演模式（见第三部分）都需要一条统一时间线

改造成本不高：`pgAutoChatOnFinish` 里"广播到 N 个 inbox"变成"append 到 timeline 一次"，`pgAutoChatDoSend` 里"合并 inbox"变成"从 lastReadId 渲染视角"。窗口自身的 `messages` 数组在自动对话模式下变成纯渲染缓存（甚至可以每次从 timeline 重建）。

## 二、逐条回应六个问题

### 1. 消息上下文膨胀

在共享时间线之上，推荐**三层结构**的上下文策略：

```
[System Prompt: 人设 + 群聊规则]
[对话摘要: "此前的对话中，陆小凤提出要去东瀛…王怜花表示…"]   ← 滚动摘要
[最近 K 条完整消息]                                        ← 滑动窗口
[本次新消息]
```

要点：

- **摘要用便宜快速的模型做**，且是异步的：当 timeline 超过阈值（如 30 条或估算 token 超过 60% context），后台触发一次摘要请求，把最老的一段折叠成一段旁白式总结。摘要期间对话正常进行，摘要完成后原子替换。
- **摘要写成"剧情旁白"而不是列表**，对角色扮演场景效果更好（"此前，众人在客栈中争论…"），模型延续性更强。
- **给用户一个可见的 token 预估条**：在群聊 Modal 顶部显示各窗口的上下文用量（比如彩色进度条 W1 ▓▓▓░ 45%），既是提醒也是很酷的可视化——用户能直观看到"这场对话烧了多少"。
- 一个偷懒但有效的兜底：**per-window 上下文条数硬上限**（如最近 40 条 + 摘要），比什么都没有强得多。

### 2. 消息嵌套 / 前缀污染

这个问题的本质是**模型模仿输入格式**。三管齐下：

1. **System prompt 明确协议**：明确告诉模型 "`[名字]:` 前缀是系统添加的路由标记，你的回复不要带任何前缀，直接说话"。加一句 few-shot 风格的反例效果显著。
2. **输出端清洗（关键且便宜）**：回复完成后，用正则剥掉开头的 `[自己名字]:` 或 `[任何名字]:` 前缀再入 timeline。模型模仿前缀非常常见，这一步能挡住 90% 的嵌套源头。
3. **引用嵌套的截断**：广播时如果检测到 content 里含有已知 agent 的 `[name]:` 模式且深度 > 1，可以选择保留（模型主动引用是合理行为）但不再递归——因为你在输出端已清洗，嵌套只会来自模型主动引用，这属于正常对话行为，不必过度处理。

顺带一提 OpenAI 的 `name` 字段：理论上 user message 支持 `name` 属性，但 TinyRouter 是透传代理，下游 provider 对 `name` 的支持参差不齐（很多直接忽略），**前缀方案反而是跨模型最稳的选择**，坚持现方案 + 输出清洗即可。

### 3. 并发安全

JS 单线程下真正的风险不是数据竞争，而是**"过期回调"（stale callback）**：定时器/Promise 完成时，世界已经变了（用户 stop 后又 start 了新一轮对话，旧定时器 fire 了）。

当前用 `abortFlag` + `isRunning` 检查基本覆盖，但有一个经典漏洞：**stop 之后立刻 start，abortFlag 被重置为 false，此时上一轮残留的 setTimeout/fetch 回调 fire，检查全部通过，污染新会话**。

标准解法是**世代号（generation/epoch token）**：

```
pgState.autoChat.session = 0;  // 每次 start 时 ++
```

所有异步回调（延迟定时器、finish hook）创建时捕获当前 session，fire 时比对：`if (capturedSession !== pgState.autoChat.session) return;`。这一个整数能替代大部分 flag 检查，且逻辑上不可能出错。强烈建议加上，成本极低。

另外建议把 `pgAutoChatOnFinish` 里的第 4、5 步（处理自己 + 触发别人）包一层 `setTimeout(..., 0)` 或 microtask，避免在 finish 的调用栈里同步链式触发多个窗口的调度——虽然当前没 bug，但把"调度"从"完成回调"中解耦后，调用栈更浅，未来加功能不容易踩坑。

### 4. 终止判断与死锁

分析你的状态机：窗口完成回复 → 广播给其他窗口 → 其他窗口 inbox 非空 → 必然会再回复。所以"所有窗口 idle 且 inbox 空"只在两种情况下发生：

- 所有窗口 `autoChatDone`（正常结束）
- **某个回复为空或失败，没有广播** → 链条断裂 → 全员沉默 → CheckAllDone 判定结束

第二种情况不是死锁而是"提前静默"，但当前它是**无声的**——对话突然结束，用户不知道为什么。建议：

- 结束时在群聊 Modal 里插入一条**系统消息**说明原因："对话已结束：所有 Agent 达到发言上限" vs "对话已结束：W2 回复失败且无人继续发言"，区分正常收尾和异常静默。
- 对失败回复提供一次**自动重试**（带退避），重试后仍失败才计数并静默。
- 加一个**看门狗兜底**：`isRunning` 期间每 30s 检查一次，如果发现状态机卡在不可能的组合（有 inbox 但没人 pending/streaming 超过 N 秒），强制触发一轮 `ProcessWindowInbox` 全扫描。这是防御未知 bug 的保险丝，不是必须但成本低。

另外注意：**iterations=0（无限）+ ≥2 窗口时对话永不自然结束**，因为每个回复必然喂给别人。这引出第三部分的"发言意愿"机制——它能让无限模式也有自然收尾。

### 5. 延迟策略

固定基数 ±50% 已经不错，但有两个更自然的升级：

- **打字模拟延迟**：延迟正比于**即将回复的上一条消息长度 + 一个阅读时间**。近似公式：`阅读时间(收到内容长度) + 基础思考时间 × random`。人不会秒回长文。
- **对话热度自适应**：维护一个 `heat` 值（最近 60s 内消息数），热度高时延迟缩短（大家在抢话），热度低时延迟拉长（冷场）。这让对话有明显的"节奏起伏"，观感提升巨大。

但真正让延迟有意义的是配套的**视觉反馈**：延迟等待期间，在群聊 Modal 和窗口标题栏显示"**王怜花正在输入…**"（带动画点点点）。这是微信/Slack 式的经典暗示，用户瞬间理解系统在做什么，等待从"卡了？"变成"哦他在打字"。这一条是本文档所有建议里**性价比最高的交互改进**。

### 6. 群聊 Modal 性能

500ms 全量 innerHTML 有三个问题：性能、**滚动位置丢失**、**文本选区丢失**（用户想复制某段话，刷新一下选区没了——这是实际体验杀手）。

方案：**基于消息 ID 的增量追加**（配合共享时间线，消息天然有稳定 id）：

- Modal 打开时记录 `renderedUpToId`
- 刷新时只 `appendChild` 新消息节点，不动已渲染的 DOM
- 正在 streaming 的消息单独处理：给它一个占位节点，只更新该节点的 textContent

而且——既然都增量了，**不如把定时刷新换成事件驱动 + streaming 直播**：窗口的流式 delta 到达时直接更新群聊 Modal 里对应消息的节点。群聊 Modal 里能看到各个 Agent 的回复**逐字打出来**，多个 Agent 同时打字时的画面感非常强，这是从"日志查看器"到"直播现场"的质变。滚动策略用聊天软件的惯例：用户在底部则自动跟随，用户上翻则停止跟随并显示"↓ N 条新消息"按钮。

## 三、让功能"眼前一亮"的扩展方向

以下按投入产出比排序，前三个都不难：

### 1. 发言意愿机制（Speak or Pass）—— 最推荐

当前最大的"不自然"是：**每个 Agent 对每条消息都必须回复**。真实群聊里，没被点名、没话可说的人会潜水。

做法：system prompt 里约定，如果 Agent 认为此刻不需要发言，输出一个特殊标记（如 `<pass/>` 或干脆空回复）。收到 pass 则不广播、不计入 replyCount（或计一半）。效果：

- 对话有了**自然的收敛**——话题聊尽时大家陆续 pass，CheckAllDone 自然触发，无限模式也能自己结束
- 出现**话题主导权流转**——某两个 Agent 就一个话题激烈讨论时，第三个会自动潜水，直到话题转到他擅长的领域
- UI 上 pass 可以显示为一个淡淡的"👀 王怜花看了一眼消息"，非常有生活感

### 2. @提及与定向发言

- 用户输入 `@王怜花 你怎么看` → 只投递给该窗口（或该窗口延迟归零、其他窗口大概率 pass）
- Agent 回复中出现 `@某人` → 被提及者的延迟缩短、发言意愿提升
- 配合发言意愿机制，对话从"全员轰炸"变成有指向性的真实群聊

### 3. 场景预设（Scenario Presets）

自动对话最好玩的用法是角色扮演和多模型辩论，但配置 4 个窗口的人设很繁琐。提供一键预设：

- **辩论模式**：正方/反方/主持人，主持人窗口 system prompt 负责总结和推进
- **圆桌评审**：用户抛代码/文案，各 Agent 扮演不同视角的评审（安全、性能、可读性）
- **剧本杀/武侠群像**：预置人设模板
- 预设本质就是一组 `{agentName, systemPrompt, 建议模型}` 的 JSON，用户选完还能改。这一个功能能把"技术 demo"变成"能玩一下午的东西"。

### 4. 导演模式（Orchestrator）—— 进阶

现在的调度是机械的"收到就（可能延迟后）回复"。可以引入一个**隐形导演**：一个廉价小模型，每当有新消息进入 timeline，它读最近几条对话，输出"下一个最该发言的是谁 + 为什么"。调度器按它的判断触发窗口。

效果是对话有了**戏剧节奏**：导演会让冲突双方连续交锋、让沉默者被 cue、在冷场时抛出转折。这就是 Generative Agents / AI Town 的核心思路，在你现有的独立迭代架构上加导演只是把"广播全员"换成"投递给导演选中的人"，改动面很小。可以做成三档调度模式让用户选：**自由发言（现状）/ 举手模式（发言意愿）/ 导演模式**。

### 5. 剧本导出与回放

共享时间线天然是一份剧本。提供：

- **导出为 Markdown**（`**王怜花**：……`格式），一键分享
- **回放模式**：对话结束后按时间轴逐条重放（可调速），配合打字动画，像看一场录播。这让好的对话结果变成**可传播的内容**——用户会截图发社交媒体，这是产品最好的自增长渠道。

### 6. 费用与用量的"仪表盘化"

自动对话是烧 token 大户，与其在 iterations=0 时放一行警告文字，不如让消耗**实时可见**：群聊 Modal 顶部一条状态栏，显示累计请求数、估算 token、各窗口上下文水位。透明本身就是最好的安全阀，而且这个仪表盘和 TinyRouter"API 代理"的产品定位天然契合——它本来就是给开发者看模型行为的工具。

## 优先级建议总结

| 优先级 | 事项 | 理由 |
|---|---|---|
| P0 | 世代号（session epoch）替代 flag 检查 | 修正确性隐患，成本极小 |
| P0 | 输出端剥离 `[name]:` 前缀 | 一行正则解决嵌套问题的主要来源 |
| P1 | 共享时间线重构 | 消灭去重/分叉/inbox 三类复杂度，为一切后续功能奠基 |
| P1 | "正在输入…"指示 + Modal 增量渲染/流式直播 | 体验质变，工作量适中 |
| P1 | 结束原因的系统消息 + 失败重试 | 消除"无声静默"的困惑 |
| P2 | 发言意愿（pass 机制）+ @提及 | 从"轮流轰炸"到"真实群聊"的关键 |
| P2 | 滚动摘要 + token 水位可视化 | 解决膨胀，顺便做出亮点 UI |
| P3 | 场景预设、导演模式、剧本导出/回放 | 把工具变成"想玩"的产品 |

核心判断：当前实现的工程质量已经过关，下一步的价值增量主要不在"修 bug"，而在**把调度从机械广播升级为有社交语义的对话编排**（发言意愿、提及、导演），以及**把不可见的过程可视化**（打字中、流式直播、token 水位）。这两条线做好，这个功能会从"多窗口自动互发"变成一个真正让人想反复把玩的多 Agent 剧场。
